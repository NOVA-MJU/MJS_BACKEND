# 통합 검색 시스템 설계 개요

이 문서는 현재 프로젝트의 Elasticsearch 기반 검색 시스템을 거시적으로 설명한다.  
핵심 목표는 여러 도메인에 흩어진 데이터를 하나의 검색 경험으로 통합하고, 한국어 검색에서 자주 발생하는 공백 차이, 부분 일치, 유사 표현, 오타 문제를 최대한 흡수하는 것이다.

## 1. 왜 통합 검색 인덱스를 따로 두었는가

이 시스템은 `Notice`, `News`, `Community`, `DepartmentSchedule`, `StudentCouncilNotice`, `Broadcast`, `MjuCalendar`처럼 성격이 다른 데이터를 검색 대상으로 삼는다.  
각 도메인의 원본 구조는 서로 다르기 때문에, 검색 시점마다 여러 인덱스를 따로 조회하고 결과를 합치는 방식은 다음 문제를 만든다.

- 쿼리 로직이 도메인별로 분산된다.
- 정렬과 랭킹 기준을 통일하기 어렵다.
- 검색 품질 개선 시 모든 도메인 쿼리를 따로 수정해야 한다.
- 검색 성능과 유지보수 비용이 같이 나빠진다.

그래서 이 프로젝트는 도메인별 인덱스를 유지하면서도, 실제 검색은 `search_unified`라는 파생 통합 인덱스에서 수행하도록 설계했다.  
즉, 도메인 인덱스는 원본에 가깝게 보존하고, 통합 인덱스는 검색 최적화에 집중하는 구조다.

## 2. 전체 구조

검색 시스템은 크게 네 단계로 나뉜다.

1. RDB의 각 도메인 데이터를 Elasticsearch 도메인 인덱스로 동기화한다.
2. 도메인 인덱스 문서를 공통 스키마의 `UnifiedSearchDocument`로 변환한다.
3. 사용자의 검색어를 정규화하고, 의도 해석과 확장어 생성을 수행한다.
4. 랭킹 정책에 따라 최종 ES 쿼리를 만들고 통합 인덱스를 조회한다.

이 구조의 장점은 "데이터 적재"와 "검색 해석"과 "랭킹 정책"을 분리했다는 점이다.  
즉, 검색 품질을 높이고 싶을 때 항상 인덱싱 로직을 다시 뜯어고칠 필요는 없고, 상황에 따라 사전, 정책, 쿼리만 조정할 수 있다.

## 3. 인덱싱 전략

### 3-1. 도메인 인덱스

각 도메인은 자신의 특성에 맞는 전처리를 거쳐 개별 Elasticsearch 문서로 저장된다.

- 공지 계열은 HTML 전처리를 수행한다.
- 커뮤니티 글은 에디터 JSON을 검색 가능한 텍스트로 변환한다.
- 그 외 데이터는 비교적 단순하게 문서화한다.

이 단계의 목적은 "원본 데이터를 검색 가능한 텍스트 자산으로 만드는 것"이다.

### 3-2. 통합 인덱스

실제 검색은 `UnifiedSearchDocument`를 기준으로 수행한다.  
이 문서에는 단순 제목과 본문뿐 아니라, 검색 품질을 위한 파생 필드가 같이 들어간다.

- `title`: 화면에 보여줄 원문 제목
- `content`: 화면에 보여줄 원문 본문
- `category`: 도메인 분류 정보
- `titleAutocomplete`: `search_as_you_type` 기반 prefix 검색용 필드
- `titleNormalized`: 공백을 제거한 제목
- `contentNormalized`: 공백을 제거한 본문
- `categoryNormalized`: 공백을 제거한 카테고리
- `searchTokens`: 형태소 기반 검색 토큰 묶음

여기서 중요한 것은 `searchTokens`와 `Normalized` 필드다.  
이 필드들이 있기 때문에 다음과 같은 검색을 더 안정적으로 처리할 수 있다.

- `해외문화탐방`으로 저장된 문서를 `해외 탐방`으로 검색
- `문화탐방`처럼 일부 핵심 단어만 입력
- 공백이 잘못 들어간 검색어
- 약한 오타가 섞인 검색어

## 4. 한국어 검색 품질을 위해 추가한 처리

### 4-1. Komoran 기반 토큰 생성

`KomoranTokenizerUtil`은 단순 자동완성용 토큰만 만들지 않는다.  
현재는 검색 품질을 높이기 위해 다음 성격의 토큰을 함께 만든다.

- 형태소 기반 핵심 명사 토큰
- 인접 단어 결합 토큰
- 공백 제거 토큰
- 중간 단어가 생략된 결합 토큰

예를 들어 `해외 문화 탐방`이라는 표현이 있으면 검색용 토큰으로 아래와 같은 값들이 같이 축적될 수 있다.

- `해외문화탐방`
- `해외 탐방`
- `해외탐방`
- `문화탐방`

이 방식은 비싼 wildcard 검색에 의존하지 않고도 부분 일치와 공백 차이를 잡기 위한 전략이다.

### 4-2. Intent Lexicon 기반 의도 해석

사용자 검색어는 단순 문자열이 아니라 "무엇을 찾고 싶은가"라는 의도로 해석된다.  
이를 위해 `intent_lexicon.json`을 두고, 특정 주제에 대해 다음 정보를 관리한다.

- 대표 intent
- 매칭 키워드
- 확장어
- 제외 키워드
- 타입별 가중치

예를 들어 `해외문화탐방` intent는 아래와 같은 표현을 하나의 검색 의도로 묶는다.

- `해외문화탐방`
- `해외 탐방`
- `해외탐방`
- `문화탐방`
- `글로벌탐방`

즉, 사용자의 입력 표현이 달라도 내부적으로는 비슷한 의도로 재해석할 수 있게 만든 것이다.

### 4-3. Compact 기준 해석

한국어 검색에서 가장 흔한 문제 중 하나는 공백이다.  
사람마다 `해외탐방`, `해외 탐방`, `해외 문화 탐방`을 다르게 입력한다.  
그래서 검색어 해석 단계에서도 원문 문자열뿐 아니라 공백 제거값을 함께 본다.

이 처리는 다음 영역에 모두 반영되어 있다.

- intent 후보 수집
- intent 점수 계산
- 오타 보정
- 실제 검색 쿼리

## 5. 검색 실행 흐름

실제 검색 요청은 대략 다음 순서로 처리된다.

1. 사용자가 `keyword`, `type`, `category`, `order`를 전달한다.
2. `SearchIntentResolver`가 검색어를 정규화하고 intent를 해석한다.
3. `SearchRankingPolicy`가 현재 운영 정책과 결합해 `SearchQueryPlan`을 만든다.
4. `UnifiedSearchQueryRepository`가 이 계획을 ES DSL로 변환한다.
5. `search_unified` 인덱스를 조회하고 결과를 응답 DTO로 바꾼다.

검색 결과가 너무 공격적인 intent 확장 때문에 비어버리는 경우를 막기 위해 fallback도 둔다.  
즉, 확장어 기반 검색이 결과를 과도하게 좁히면, intent 확장을 제거한 계획으로 한 번 더 조회할 수 있게 만들었다.

## 6. 랭킹 설계

이 시스템의 랭킹은 "하나의 점수"가 아니라 여러 신호의 합으로 구성된다.

### 6-1. 텍스트 관련 점수

기본적으로 아래 신호를 함께 사용한다.

- 제목 정확 일치
- 제목 공백 제거 일치
- 검색 토큰 일치
- 자동완성 prefix 일치
- 카테고리 일치
- 오타 fuzzy 일치
- 확장어 일치

이 중 제목 exact match와 compact match는 강한 신호다.  
즉, 사용자가 찾은 핵심 표현이 제목에 그대로 있으면 가장 높은 점수를 받도록 설계했다.

### 6-2. 도메인 우선순위

학교 검색에서는 모든 문서가 동일하게 중요하지 않다.  
예를 들어 학생이 어떤 제도를 검색할 때는 커뮤니티 글보다 공지나 학사 일정이 먼저 나와야 할 가능성이 높다.

그래서 intent별로 타입 가중치를 줄 수 있게 만들었다.

- `NOTICE`
- `STUDENT_COUNCIL_NOTICE`
- `MJU_CALENDAR`
- 그 외 도메인

이 가중치는 `intent_lexicon.json`에서 주제별로 다르게 조정한다.

### 6-3. 최신성 / 인기 보정

검색은 항상 텍스트 일치도만으로 끝나지 않는다.  
같은 주제라면 너무 오래된 문서보다 최근 문서가, 비슷한 커뮤니티 글이라면 반응이 많은 글이 더 유용할 수 있다.

그래서 정책 파일에서 아래 보정을 따로 관리한다.

- 최신성 boost
- 좋아요 수 boost
- 댓글 수 boost
- intent 검색 시 최근 기간 제한

이 값은 `search_ranking_policy.json`과 runtime policy를 통해 운영 중 조정할 수 있다.

## 7. 정책 파일 중심 운영

랭킹 관련 수치를 코드에 박아두면 배포 없이 튜닝하기 어렵다.  
그래서 현재 시스템은 주요 랭킹 수치를 정책 파일로 분리했다.

정책에서 관리하는 대표 값은 다음과 같다.

- `expansionTermBoost`
- `autocompleteBoost`
- `exactTitleMatchBoost`
- `compactTitleMatchBoost`
- `searchTokenMatchBoost`
- `typoFuzzyBoost`
- `categoryMatchBoost`
- `noticeTypeBoost`
- `noticeGeneralCategoryBoost`

이 구조 덕분에 검색 로그를 보고 점수를 조정할 때, 검색 로직 전체를 다시 설계하지 않고도 반복적인 튜닝이 가능하다.

## 8. Redis의 역할

현재 통합 검색 결과 자체를 Redis에 캐시하고 있지는 않다.  
검색 결과는 매 요청마다 Elasticsearch의 `search_unified` 인덱스를 직접 조회한다.

Redis는 검색 응답 캐시가 아니라 실시간 검색어 집계 용도로만 사용한다.

- 사용자가 검색한 키워드 기록
- 인기 검색어 집계
- 오래된 검색 기록 TTL 정리

즉, 검색 결과가 계속 같게 보이는 원인을 Redis 캐시로 볼 근거는 현재 구조에는 없다.  
응답이 동일하다면 우선 확인해야 할 것은 아래다.

- 실제 통합 인덱스가 재생성되었는지
- 검색 쿼리 정책이 여전히 같은지
- 기간 필터가 결과를 잘라내고 있지 않은지
- 대상 문서가 실제로 통합 인덱스에 들어갔는지

## 9. 기간 필터와 비즈니스 전략

검색에서는 최신성이 중요할 때가 많다.  
특히 학생들이 짧게 `장학금`, `수강`, `해외탐방`처럼 검색하는 경우, 아주 오래된 공지보다 최근 공지가 먼저 보이는 것이 일반적으로 더 유용하다.

그래서 이 시스템은 intent 확장 검색일 때 최근 문서에 가중치를 두는 정책을 사용한다.  
다만 이 정책을 모든 검색에 동일하게 적용하면, `2025학년도 동계 전공학문연계 해외문화탐방 참가자 선발안내`처럼 긴 제목형 검색에서 오히려 원하는 문서가 탈락할 수 있다.

현재 전략은 아래처럼 가져간다.

- 짧고 의도 중심인 검색어: 최신성 필터와 최신성 boost를 적극 적용
- 긴 제목형 검색어: 기간 필터를 완화하거나 적용하지 않음
- 결과 정렬은 relevance를 기본으로 하되, 최신성은 보조 신호로만 사용

비즈니스 관점에서 보면 이 전략은 두 사용 시나리오를 분리한 것이다.

- 탐색형 검색: 지금 중요한 최신 공지를 빠르게 찾게 한다.
- 지목형 검색: 사용자가 이미 제목이나 문구를 알고 있을 때 해당 문서를 정확히 찾게 한다.

이 둘을 구분하지 않으면 검색이 항상 최신성에 끌리거나, 반대로 너무 오래된 문서까지 넓게 섞여 품질이 무너질 수 있다.

## 10. 최근 튜닝 내용

최근 검색 품질 개선은 아래 원칙으로 진행했다.

- 통합 인덱스에 `titleNormalized`, `contentNormalized`, `categoryNormalized`, `searchTokens`를 추가
- 공백 제거와 복합명사 결합 토큰을 인덱싱 단계에서 미리 생성
- intent lexicon에 `해외문화탐방` 계열 표현을 추가
- 긴 질의에 대해서는 `title`과 `searchTokens` 중심의 stricter match를 추가
- 짧은 intent 질의에만 최근 기간 필터를 적용하도록 조정
- 본문 일치 점수는 낮추고 제목/핵심 토큰 일치 점수는 높임

이 튜닝의 목적은 단순하다.

- 긴 제목 전체를 넣으면 해당 문서를 우선 찾게 한다.
- `해외탐방`, `문화탐방`처럼 줄여 검색해도 관련 공지를 잡게 한다.
- 본문에 단어 몇 개가 들어간 일반 공지가 상단을 점령하지 못하게 한다.

## 11. 성능 측면에서의 설계 의도

현재 구조는 성능을 위해 다음 원칙을 따른다.

- 검색 시 여러 도메인 인덱스를 동시에 조회하지 않고 통합 인덱스 하나를 조회한다.
- wildcard 중심이 아니라 미리 생성된 검색 토큰과 정규화 필드를 사용한다.
- 공백 처리와 부분 일치를 인덱싱 시점으로 일부 이동시킨다.
- intent, 정책, 쿼리 생성 책임을 분리해 불필요한 복잡도를 줄인다.

즉, "검색 때마다 복잡하게 계산하는 방식"보다 "인덱싱 때 준비하고 검색 때 빠르게 활용하는 방식"에 가깝다.

## 12. 운영 시 주의할 점

이 시스템은 통합 인덱스 매핑이나 검색 토큰 생성 규칙이 바뀌면 재색인이 필요하다.  
특히 다음이 바뀌었을 때는 `search_unified`를 다시 만드는 것이 안전하다.

- `UnifiedSearchDocument` 필드 구조
- `KomoranTokenizerUtil` 토큰 생성 규칙
- 도메인 문서 -> 통합 문서 매핑 규칙

반대로 아래 항목은 재배포 또는 설정 반영만으로 조정 가능한 경우가 많다.

- intent 사전 추가
- 확장어 추가
- 타입별 가중치 조정
- 랭킹 boost 값 조정

## 13. 이 검색 시스템의 핵심 철학

이 검색 시스템은 단순히 "문자열 포함 여부"를 보는 시스템이 아니다.  
학교 서비스 검색에서 중요한 것은 사용자가 정확한 용어를 몰라도 원하는 정보를 찾을 수 있게 만드는 것이다.

그래서 현재 구조는 다음 철학 위에 서 있다.

- 데이터는 도메인별로 보존하되 검색은 통합해서 본다.
- 한국어 공백 차이와 표현 차이를 기본 문제로 간주한다.
- 검색 품질은 코드가 아니라 정책과 사전으로 계속 개선할 수 있어야 한다.
- 빠른 검색과 좋은 랭킹을 위해 인덱싱 단계에서 최대한 준비한다.

정리하면, 이 시스템은 "통합 인덱스 + 정규화 필드 + 형태소 기반 토큰 + intent 사전 + 정책 기반 랭킹"의 조합으로 구축된 검색 시스템이다.

## 14. 현재 랭킹 운영 원칙

현재 운영 기준에서 가장 중요한 원칙은 "결과를 기간으로 잘라내지 않는다"는 점이다.  
이전에는 intent 확장 검색에서 최근 기간을 기준으로 결과를 제한하는 방식도 고려했지만, 긴 제목형 검색에서는 원하는 과거 공지가 아예 탈락하는 문제가 있었다.

그래서 현재는 아래 순서로 랭킹을 해석한다.

- 1차: 제목 exact match, compact title match, search token phrase match 같은 강한 텍스트 신호
- 2차: intent 확장어, 카테고리 일치, 자동완성, 오타 보정 같은 보조 신호
- 3차: 최신성, 좋아요 수, 댓글 수 같은 후순위 신호

즉, 비즈니스 전략은 아래와 같다.

- 사용자가 제목을 거의 알고 검색하면 해당 문서를 가장 먼저 보여준다.
- 사용자가 주제만 알고 탐색형으로 검색하면 관련도 높은 최신 공지를 더 우선한다.
- 비슷한 내용의 공지가 여러 학기에 반복되면, 관련도가 거의 같을 때 더 최근 공지가 앞에 오도록 한다.

정렬 측면에서는 `relevance`를 기본으로 사용하되, 내부적으로 최신성 boost와 `date desc` 보조 정렬을 함께 사용한다.  
그래서 완전히 최신순으로 밀어붙이지 않으면서도, 같은 주제의 반복 공지 중에서는 최근 공지가 우선되는 구조다.
